## 问题描述

Agent DAG 执行完毕后，主仓库目录下没有任何文件，只有隐藏的 `.git` 和 `.worktrees` 目录。Agent 产出的所有文件仅存在于各自的 worktree 子目录中（`.worktrees/<agent-id>/`），从未被 commit 和 merge 回主分支。

当前流程缺失两个关键环节：

1. **Agent 完成后未 commit**：Agent 在 worktree 中写入的文件只是 uncommitted 工作区改动。如果 worktree 被清理，这些文件直接丢失。
2. **Mission 完成后未 merge**：所有 agent 分支（`agent/<agent-id>`）的改动从未被合并到主分支（`main`）。用户在主仓库目录下看不到任何产出。

## 复现步骤

1. 创建一个 Mission 并 Plan 出若干任务
2. 点击 Start Mission，选择工作区目录
3. 等待 Mission 执行完成（状态变为 completed）
4. 在工作区目录下执行 `ls`
5. 发现只有 `.git/` 和 `.worktrees/`，没有任何 Agent 产出的文件

## 期望行为

### Agent 完成时（单任务粒度）

- Agent 完成后，应将 worktree 内所有改动自动 `git add -A && git commit`
- commit message 应包含任务标题（如 `[Task] Implement user auth module`）
- 确保 `agent/<agent-id>` 分支上有一个完整的 commit 记录

### Mission 完成时（全局粒度）

- 当 Mission 达到终态（completed）时，应按 DAG 拓扑顺序将所有成功任务的 agent 分支 merge 到主分支：
  - 依赖在前的先 merge，依赖在后的后 merge
  - 无依赖关系的按完成时间顺序 merge
  - merge 失败（冲突）时标记为需要用户手动解决，并发出事件通知
- merge 完成后清理 worktree 和对应的 agent 分支
- Mission 失败时（部分任务 failed）不自动 merge，保留 worktree 供排查

### 事件通知

| 事件名 | 时机 | payload |
|--------|------|---------|
| `agent-committed` | Agent 产出被 commit 后 | `{ agent_id, task_id, commit_hash, branch }` |
| `mission-merge-progress` | 每个分支 merge 后 | `{ mission_id, branch, status: "merged" \| "conflict" }` |
| `mission-merge-completed` | 全部 merge 完成 | `{ mission_id, total_merged, conflicts }` |

## 建议修改方案

### Phase 1: Agent 自动 commit（最小可行）

在 `scheduler.rs` Agent 完成回调中：

```rust
// Agent 完成后，在 worktree 中 commit 所有改动
if task_status == "completed" {
    let wt_manager = WorktreeManager::new(repo_path.clone());
    wt_manager.commit_worktree(&aid, &task_title)?;
}
```

`WorktreeManager::commit_worktree` 实现：
- `git add -A` (stage all)
- `git commit -m "[Task] {task_title}"`
- 返回 commit hash

### Phase 2: Mission merge

在 `check_mission_terminal` 返回 `"completed"` 时触发：

1. 查询 DAG 拓扑序的已完成任务列表
2. 按顺序对每个 `agent/<id>` 分支执行 `git merge --no-ff` 到 main
3. 遇到冲突时记录并跳过，后续通知用户
4. 清理已成功 merge 的 worktree 和分支

### Phase 3: 冲突处理 UI

在前端 WorkspaceView 中展示 merge 冲突，提供手动解决入口。

## 优先级评估

**Critical** — 此问题导致 Agent 的所有工作产出对用户不可见，是 Mission 执行流程的核心缺陷，直接阻塞产品可用性。Phase 1（自动 commit）应立即修复，Phase 2（merge）应紧随其后。
