# DTS — Defect Tracking Sheet 规范

## 编号规则

```
dts-{YYYY}-{MM}-{DD}-{6位hex}
```

- 日期为**提交日期**
- hex 部分使用随机生成的 6 位十六进制字符，确保同日内唯一
- 示例：`dts-2026-04-02-a3f7e1`

## 文件命名

每个问题单对应一个独立 markdown 文件，放在 `docs/dts/` 目录下：

```
docs/dts/dts-2026-04-02-a3f7e1.md
```

## 模板

```markdown
# DTS-{编号}

> **模块**: {FM-XX 模块名}  
> **严重级别**: {Critical | Major | Minor | Cosmetic}  
> **状态**: {Open | In Progress | Fixed | Closed | Won't Fix}  
> **提交日期**: {YYYY-MM-DD}  
> **报告人**: {用户 | 开发者}

---

## 问题描述

{简明扼要地描述问题现象，1-3 段即可}

## 复现步骤

1. {步骤 1}
2. {步骤 2}
3. ...

## 期望行为

{描述正确的行为应该是什么}

## 关联代码

| 文件 | 说明 |
|------|------|
| `path/to/file` | 简要说明 |

## 修复记录

> 此节在修复完成后由开发者填写，提交前留空

**修复日期**: {YYYY-MM-DD}

| 文件 | 变更 |
|------|------|
| `path/to/file` | 变更说明 |

## 优先级评估

{1-2 句话说明影响范围和紧急程度}
```

## 严重级别定义

| 级别 | 定义 | 响应要求 |
|------|------|----------|
| **Critical** | 核心流程不可用，数据丢失风险，无 workaround | 立即修复 |
| **Major** | 主要功能受损，但有 workaround 或不影响数据安全 | 当前迭代内修复 |
| **Minor** | 非核心功能异常，体验欠佳 | 排入后续迭代 |
| **Cosmetic** | 界面、文案等纯表现层问题 | 有空再修 |

## 状态流转

```
Open → In Progress → Fixed → Closed
                  ↘ Won't Fix
```

- **Open**: 已记录，待处理
- **In Progress**: 开发者正在修复
- **Fixed**: 开发者完成修复，待用户验证
- **Closed**: 用户确认修复，问题关闭
- **Won't Fix**: 经评估不修复（需注明理由）

## 注意事项

- **状态变更权限**：`Open → In Progress → Fixed` 由开发者操作；`Fixed → Closed` 由报告人（用户）确认后操作
- 修复记录应包含变更的文件列表和简要说明，方便 code review
- 问题单创建后编号不可变更
