## 问题描述

创建 Mission 后点击 "Confirm & Start"，弹出系统文件选择器（file-picker）要求用户选择一个 Git 仓库目录。这个设计存在以下问题：

1. **场景不合理**：用户通常需要新建一个空目录供 Agent 执行任务，而非选择已有仓库。选择空目录后报错 `'xxx' is not a git repository`，体验割裂。
2. **Mission 状态卡死**：confirm 已将 Mission 从 `draft` 推进到 `planned`，但后续 `startMissionExecution` 失败后，虽然 Mission 仍为 `planned` 可重试，但用户缺乏明确指引，感知上 Mission "无法操控"。
3. **缺少默认位置**：每次启动 Mission 都要手动选择目录，没有提供合理的默认值。

## 复现步骤

1. 创建一个包含任务的 Mission
2. 点击 "Confirm & Start"
3. 在弹出的 file-picker 中选择一个空目录（非 git 仓库）
4. 观察到错误提示 `'xxx' is not a git repository`
5. Mission 停留在 `planned` 状态，用户不清楚下一步操作

## 期望行为

启动 Mission 时应提供**智能默认 + 用户可选**的工作区策略：

1. **提供默认工作区路径**：例如 `~/miragenty-workspaces/<mission-slug>/`，自动创建并 `git init`
2. **启动确认对话框**提供两个选项：
   - **使用默认位置**：系统自动创建目录 + `git init`，零操作启动
   - **选择自定义目录**：弹出 file-picker，选择后若非 git 仓库则自动执行 `git init`（需明确提示用户）
3. **错误恢复**：启动失败后应有明确的重试入口和错误原因说明

## 建议修改方案

### 后端

- `start_mission_execution` 增加逻辑：若 `repo_path` 不是 git 仓库，自动执行 `git init` 而非拒绝
- 新增配置项 `default_workspace_root`（默认 `~/miragenty-workspaces/`）
- 自动生成工作区路径：`{default_workspace_root}/{mission-title-slug}-{short-id}/`

### 前端

- 将当前的 file-picker 替换为自定义的**启动确认对话框**（modal），包含：
  - 默认路径预览（可编辑）
  - "使用默认位置" 主按钮
  - "选择其他目录" 次按钮
- 错误状态下的 Mission 卡片增加 "重试启动" 入口和错误原因
